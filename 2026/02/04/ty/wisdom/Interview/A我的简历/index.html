模板 2：详细技术版（适合面试展开讲）

一、缓存层引入

  • 业务背景：老人信息查询接口日均调用量较大，原有查询依赖数据库直连，响应时间较长
  • 技术方案：引入 Redis 作为分布式缓存层，结合本地 Caffeine 构建二级缓存，对缓存 Key 进行规范化设计（业务前缀:数据类型:主键:版本）
  • 技术细节：使用 Cache-Aside 模式，读时回种 + 更新时删除；TTL 设置 12 小时并加随机偏移，防止缓存雪崩
  • 成果：核心查询接口 P99 响应时间从 500ms 降至 50ms（基于 Postman 测得）

二、RBAC 权限中心配置

  • 业务背景：原有系统采用硬编码角色判断，扩展性差且存在越权风险
  • 技术方案：基于 Spring Security 框架，搭建五张表（user、role、power、user_role、role_power）的 RBAC 模型
  • 技术细节：登录时通过 UserDetailsService 从 DB 加载用户权限链；URL 级用 Filter 拦截，方法级用 @PreAuthorize 注解
  • 粒度设计：覆盖 URL 级、方法级、数据级三种粒度；权限修改通过 Redis 缓存失效实现实时生效
  • 成果：覆盖 10+ 功能模块的权限隔离

三、MySQL 索引优化

  • 业务场景：老人健康数据查询（按心率、血氧等体征过滤）
  • 优化思路：通过 EXPLAIN 分析慢 SQL，针对性调整索引设计；遵循最左前缀原则，避免范围查询后字段索引失效
  • 成果：关键数据查询响应时间从约 100ms 降至 30~40ms

四、IoT 设备接入协助

  • 业务背景：平台需对接老人可穿戴设备（手环、血压计等），采集心率、血氧、运动数据
  • 协助内容：参与设备数据上报接口调试、数据解析与存储逻辑开发，保障数据 10 分钟级实时同步

常见面试问题（含标准答案）

Q1：请介绍一下你的实习经历？（高频题）

答题思路：按 STAR 法则，但简化版——背景 1 句话、职责 1 句话、成果 3 条。

参考答案：
“我在 2025 年暑期在乐鱼健康实习了三个月，这是一家做养老机构智慧管理平台的公司。
我主要参与后端开发，负责缓存层引入、权限中心配置和 MySQL 索引优化三块工作。
成果上，核心查询接口响应时间从 500ms 降到 50ms，RBAC 权限模块覆盖了 10+ 功能模块，关键 SQL 查询从 100ms 优化到 30~40ms。”

Q2：你在实习中遇到过最大的技术挑战是什么？怎么解决的？（必问）

真实回答模板（基于真实踩坑）：

“最大的挑战是 MySQL 索引优化。刚开始我按照 WHERE 条件里出现的字段建了联合索引 (老人, 心率, 时间)，但后来通过 EXPLAIN 发现这个索引根本没被用到——因为 SQL 是 WHERE 心率 > 110，没有用到最左边的’老人’字段。

解决方法是重新设计索引：把范围字段’心率’放在第一位，’时间’放后面用于排序覆盖；并且把区分度低的’老人姓名’字段从索引里去掉。优化后接口响应时间从 100ms 降到 30ms 左右。”

Q3：Redis 缓存和数据库一致性怎么保证？（高频题）

答题思路：先说 Cache-Aside，再说延迟双删，最后说适用场景。

参考答案：
“我们用的是 Cache-Aside 模式：读时回种缓存，写时先更新数据库再删除缓存。

为了解决’读旧值’问题（更新完数据库但还没删缓存时，其他线程读到旧缓存），我们用了延迟双删：第一次删缓存 → 写数据库 → 延迟 500ms 后再删一次缓存。

这是最终一致性方案，适合养老业务对实时性要求不高的场景。如果业务要求强一致，那就不能用缓存，得用分布式锁或者直接读数据库。”

Q4：Spring Security 的权限校验在哪一层？（必问）

答题思路：Filter + @PreAuthorize 双层防护。

参考答案：
“我们用了 Filter + @PreAuthorize 注解双层方案。

Filter 层做 URL 级别的粗粒度拦截，登录态校验、URL 权限判断都在这一层。
@PreAuthorize 注解做方法级别的细粒度校验，比如 @PreAuthorize(“hasAuthority(‘elder:delete’)”)。

底层用的是 RBAC 模型，数据库里建了 user、role、power 三张主表加两张关联表，登录时通过 UserDetailsService 把权限集合查出来交给 Spring Security 校验。”

Q5：ThreadLocal 在异步场景下会丢失吗？怎么解决？（高频加分题）

答题思路：会丢失，给出三种方案。

参考答案：
“会丢失。ThreadLocal 是线程本地变量，异步任务跑在线程池的另一条线程上，那条线程的 ThreadLocal 是空的，所以取不到用户信息。

解决方案有三种：

  • 一是手动传参，把 userId 作为方法参数传过去，最简单但侵入性强；
  • 二是用 Spring 的 TaskDecorator，在任务执行前把主线程的 ThreadLocal 复制到新线程；
  • 三是用阿里的 TransmittableThreadLocal 框架，无侵入地自动传递 ThreadLocal 值。”

实习中遇到的挑战（基于真实踩坑）

挑战 1：MySQL 联合索引违反最左前缀

问题：实习初期，我按 WHERE 条件字段直接建了联合索引 (老人, 心率, 时间)，但实际 SQL 是 WHERE 心率 > 110，导致索引完全没被用到。

踩坑过程：

  • 第一版索引：(老人, 心率, 时间) → EXPLAIN 显示 type=ALL，全表扫描
  • 排查：用 EXPLAIN ANALYZE 看实际执行计划，发现 key 列为 NULL
  • 根因：违反了最左前缀原则，SQL 没用到’老人’字段

解决方案：

  • 重建索引：(心率) 单列索引，或 (心率, 时间) 用于覆盖排序
  • 区分度低的字段（姓名）不进索引
  • 任何索引设计先用 EXPLAIN 验证

收获：理解了索引的”最左前缀”不是语法，而是 B+ 树的物理结构决定的。

挑战 2：Redis Key 设计不规范

问题：刚开始缓存 Key 就是 elderInfo:1001 这种简单拼接，没有版本号，导致缓存结构升级时（旧字段变更）无法强制刷新旧缓存。

踩坑过程：

  • 上线了一个新字段（紧急联系人电话）后，部分用户反馈看不到
  • 排查发现：老用户缓存的 JSON 里没有这个字段，但缓存没过期
  • 根因：Key 设计没版本号，无法强制刷新

解决方案：

  • 引入版本号：elder:info:{elderId}:v2
  • 任何缓存结构变更 → 版本号 +1 → 老 Key 自然过期
  • 配合 TTL 随机偏移（10~14h）防止雪崩

收获：缓存 Key 设计要考虑”未来变更兼容性”，版本号是必须的。

挑战 3：Spring Security 权限”强制重新登录”的设计失误

问题：实习初期，权限修改后我设计的是”强制用户重新登录”才能生效，结果被组长 review 打回。

踩坑过程：

  • 第一版设计：权限修改 → DEL 用户 Session → 用户必须重新登录
  • 问题：医生正在用系统查数据，运营改了权限被踢下线，影响业务
  • 根因：没考虑”在线用户实时生效”的需求

解决方案：

  • 把用户权限缓存到 Redis（Key: user:authority:{username}）
  • 权限修改时 DEL 缓存
  • 用户下次请求时自动重新加载，全程无感知
  • 或者用 Spring Security 的 SessionRegistry 主动失效会话

收获：权限系统的设计要考虑”实时性 vs 一致性”的 trade-off，不能简单粗暴”踢人下线”。

挑战 4：VSCode 写 Java 的工具链问题

问题：入职时组长让我用 VSCode 写 Java，但我发现 VSCode 的 Java 体验比 IDEA 差很多。

踩坑过程：

  • VSCode 装 Java 扩展包（Extension Pack for Java）后能用，但智能提示、重构、调试体验都不如 IDEA
  • 团队其他人都是 IDEA，于是我也切换到了 IDEA
  • IDEA 装了 MyBatisX 插件（Mapper 接口跳 XML）、Chinese Language Pack（汉化）

解决方案：

  • 切换到 IntelliJ IDEA（社区版免费）
  • 装好 MyBatisX / Free MyBatis plugin、Lombok、Alibaba Java Coding Guidelines 等插件
  • 团队统一工具链，减少沟通成本

收获：工具链要服从团队规范，不要盲目追求”新潮”。

挑战 5：IoT 设备数据接入的实时性保障

问题：IoT 设备每 10 分钟上报一次数据，但偶尔会出现数据丢失或延迟。

踩坑过程：

  • 第一版：设备直接 HTTP POST 到后端 → 后端写库
  • 问题：设备网络不稳定时数据丢失；高峰期接口被打爆

解决方案：

  • 设备侧加入重试机制（指数退避）
  • 后端引入消息队列（RabbitMQ）做削峰填谷
  • 数据库采用时序表设计（按时间分区）
  • 加入数据质量监控（缺失告警）

收获：IoT 系统的设计要考虑”设备不可靠 + 网络不稳定”的现实约束，不能假设理想环境。
