SpringCloud概述

我自己理解的就是多个SpringBoot结合一下,怎么结合就是通过SpringCloud,
比如之前做的养老服务,可以将整个单体项目拆分为一个一个的,支付作为一个服务,老人信息管理作为一个服务

项目结构

  1. 总项目
    1. 具体模块1(服务模块)
      1. 具体服务1(商品服务)
      2. 具体服务2(订单服务)
    2. 具体模块2(类模块)
      1. 具体类1(商品类)
      2. 具体类2(订单类)

Nacos 注册中心

那么不同服务是怎么进行联系的呢,就是通过nacos这个注册中心进行的,将nacos比作一个小区管理者,
你加入这个小区(项目),那么就要先去小区管理者(nacos)那里进行注册,然后之后有人找你(被调用)的时候,
就在小区管理者(nacos)里面寻找,找到了说明你以及加入这个小区(项目)了

服务查找

当一个服务需要寻找其他服务的时候,就要通过服务查找进行

具体操作如下:

  1. 通过DiscoverClient类创建对象discoverClient
  2. 通过discoverClient.getServices()方法可以获得所有已注册服务
  3. 再通过discoveryClient.getInstances(具体服务名(spring.application.name))
    得到此服务的所有ip(getHost())和端口(getPort())的列表

负载均衡

为什么需要负载均衡?
因为如果上一步那样,那么只有当这个服务对应的服务器出问题(宕机…)之后,才能使用其他服务器,
这样的话资源分配不均衡,会造成资源浪费,所以需要负载均衡
现在一般有两种实现负载的方式,这里讲简单的一种,另一种见
(https://www.bilibili.com/video/BV1UJc2ezEFU?spm_id_from=333.788.videopod.sections&vd_source=5995638132315e6a882e78ef30796cff&p=10)

具体实现:
在service服务里面的configRestTemplate上面加上@LoadBalanced注解,就可以自动实现负载均衡,
在这个注解里面已经配置好了负载均衡算法

经典面试题目

如果注册中心宕机,远程调用还能实现吗?

答: Nacos客户端启动时，会拉取所有服务的实例列表并缓存。
只要在缓存有效期内（即使Nacos宕机），所有服务都能调用，
无论之前是否调用过

Nacos 配置中心

如何使用

  1. 需要引入依赖

  2. 在nacos里面新建配置 service-order.properties,配置格式选择Properties,编写配置内容

  3. 在微服务里面的配置文件(application.properties)里面加入
    spring.config.import=nacos:service-order.properties

  4. 得到对应配置的值

    1. 创建properties.java类 
    2. @Component
@ConfigurationProperties(prefix = "order")
@Data
public class OrderProperties {
private String timeout;
private String autoConfirm;
}
    3. @ConfigurationProperties(prefix = “order”)是自动刷新配置的注解,
      prefix是前缀,配置内容里面的-c转为驼峰命名法的大写作为属性名
    4. 然后在其他类里面通过OrderProperties类进行操作

经典面试题目

如果配置中心和application.properties有相同配置,谁的优先级更高?

答: 从设计的角度来说,配置中心起到统领全局的作用,所以的配置中心的更高,
从配置中心如果引入多个配置,优先引入的配置优先级更高

Nacos 数据隔离

因为不同环境下面不同的配置文件不一样,比如dev(开发),test(测试),prod(生产)三个环境,
开发里面要求timeout=30s,但是生产里面可能就是timeout=30min这样,
还有order(订单服务)需要一个common.properties,product(产品服务)也需要同名的配置文件,
那么这个该怎么区分?所以衍生出来了数据隔离

Nacos操作

  1. 在命名空间列表下里面创建三个命名空间(根据实际业务),分为dev,test,prod
  2. 在创建配置时注意对应的命名空间,还有分组(order组,product组,,,)
  3. 在各自的服务配置文件里面导入对应的配置文件
  4. 可以将properties文件变为yml文件,更好看一点
  6. 根据不同的环境,选择不同的命名空间即可,赋值部分:后是默认值(如果前提赋值为空)
