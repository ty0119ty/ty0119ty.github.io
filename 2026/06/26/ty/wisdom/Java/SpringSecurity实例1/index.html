SpringSecurity实例1

发表于2026-06-26|更新于2026-06-26|八股Java
|浏览量:

前言

学 Spring Security 这块一直挺头疼的,光”认证”和”授权”两个词就够绕一阵,
而且网上教程上来就一堆概念,代码又跳来跳去,看完整个人都是懵的。

这篇文章就用一个最简单、最完整的 RBAC 例子,把 Spring Security 的核心流程
跑一遍——认证怎么走、URL 权限怎么配、方法级权限怎么用、多角色怎么并存,
全部用一个能 mvn spring-boot:run 直接跑起来的 demo 来讲清楚。

本文基于 Spring Boot 3.2.0 + Spring Security 6.x + Java 17,
后续版本 API 可能略有调整,请以官方文档为准。

准备

技术栈

  • Spring Boot 3.2.0
  • Spring Security 6.x(随 Boot 走)
  • Thymeleaf(前端模板,带 SpringSecurity 扩展)
  • JDK 17

pom.xml 关键依赖

项目结构

核心实现

1. SecurityConfig.java(URL 级别权限)

这是整个 demo 最核心的文件,所有 URL 权限规则都从这里配。

几个关键点(后面”补充知识点”里会展开):

  • hasRole("VIP1") = hasAuthority("ROLE_VIP1")(自动加 ROLE_ 前缀)
  • defaultSuccessUrl("/", true) 第二个参数 true = 强制跳首页,不让用户去访问他”被拦截前的 URL”(防开放重定向漏洞)
  • csrf().disable() 演示项目常用,生产要看场景

2. InMemoryUserService.java(用户服务)

**这里实现了 Spring Security 最关键的接口 UserDetailsService**——
认证时 Spring Security 会回调这个接口的 loadUserByUsername(),你只要返回”用户名+密码+权限列表”就行。

重点:SimpleGrantedAuthority 接收的字符串要带 ROLE_ 前缀,
因为 hasRole("VIP1") 内部就是拿 "ROLE_VIP1" 去匹配的。
这一步错了,鉴权就直接失败,登录后进不去任何受保护页面。

3. User.java(简化实体)

简化版,只用 HashSet 装角色字符串,生产里肯定要换成数据库表(用户表 + 角色表 + 用户角色关联表)。

4. Vip1Controller.java + @PreAuthorize

方法级权限:除了 URL 规则,还可以在 Controller 类/方法上加 @PreAuthorize,做更细粒度的控制。

注意 @PreAuthorize("hasRole('VIP1')") 里直接写 'VIP1' 就行(不用带 ROLE_),
因为 Spring Security 在解析注解时也会自动加 ROLE_ 前缀——和 hasRole() 行为一致。

Vip2Controller / Vip3Controller 结构完全一样,只是 @PreAuthorize 里的角色不同。

5. HomeController.java

注意 Authentication 是 Spring Security 的对象,由框架在请求进入 Controller 时注入;
里面包含用户名(getName())、权限列表(getAuthorities())等。

权限说明(用户多角色,不是角色继承)

重要:Spring Security 默认没有”角色继承”机制——
user2 能进 /vip1/** 不是因为 VIP2”继承”了 VIP1,而是因为 user2 这个用户同时被分配了 VIP1 和 VIP2 两个角色。
这是两件不同的事,后面”补充知识点”会详细讲两种实现方式。

整体流程图

补充知识点

1. ROLE_ 前缀与 hasRole / hasAuthority 的关系(高频坑)

Spring Security 里两种鉴权 API,必须分清:

写法 匹配 GrantedAuthority 里的字符串
hasRole("VIP1") ROLE_VIP1(自动加 ROLE_ 前缀)
hasAuthority("ROLE_VIP1") ROLE_VIP1(精确匹配,不补前缀)
hasAuthority("VIP1") VIP1(精确匹配)

所以:

  • 数据库/代码里存的角色字符串,要么统一存 "ROLE_VIP1"(配合 hasRole),要么统一存 "VIP1"(配合 hasAuthority),别混着用
  • 混用的结果:**登录成功,但所有受保护页面都进不去,跳 /denied**——调试半天发现是这里

2. csrf().disable() 为什么要关 / 生产要不要开

  • CSRF = Cross-Site Request Forgery,跨站请求伪造。简单说:用户登录了银行网站,
    没退出又访问了恶意网站,恶意网站用用户的 cookie 偷偷发起转账请求。
  • Spring Security 默认开启 CSRF 防护:所有写操作(POST/PUT/DELETE)要求带一个 CSRF token,否则拒绝。
  • 演示项目关掉的原因:浏览器表单直接 POST 就行,不写 CSRF token 也能跑
  • 生产建议:
    • 传统服务端渲染(Thymeleaf/JSP)→ 开启,框架自动在表单里塞 token
    • 前后端分离 + JWT → 关掉(因为前端用 Authorization 头,不是 cookie,不需要 CSRF)
    • 纯 API 服务 → 关掉

3. 权限继承的两种实现方式

方式 1:用户分配多个角色(本 demo 用)

优点:简单直接;缺点:用户多了角色容易乱(用户 100 个,每个人都有 5 个角色 → 表里 500 行)

方式 2:RoleHierarchy(真正的”角色继承”)

之后用户只需要一个角色 ROLE_VIP3,就能自动继承 VIP2、VIP1 的所有权限——不需要在用户上分配多个角色。
优点:角色关系清晰;缺点:配置略复杂,继承链一长容易乱。

本 demo 用方式 1(用户多角色)。生产里两种都用得到:RBAC 模型里通常 ADMIN > 普通用户,
用方式 2 配层级;同一个用户有多个角色(既是 VIP 又是 SVIP)用方式 1。

4. @EnableMethodSecurity vs @EnableGlobalMethodSecurity

  • @EnableMethodSecurity:Spring Security 5.6+ 推荐,本 demo 用这个
  • @EnableGlobalMethodSecurity:Spring Security 老版本用法,已弃用
  • 区别:
    • 新版默认 prePostEnabled = true,直接支持 @PreAuthorize / @PostAuthorize
    • 老版需要 @EnableGlobalMethodSecurity(prePostEnabled = true) 显式开启

看到老教程写 @EnableGlobalMethodSecurity 的,直接换新写法就行。

5. defaultSuccessUrl("/", true) 的 true 含义

  • defaultSuccessUrl("/", false)(默认):登录成功后,如果用户是因为访问某个受保护 URL 被拦截而登录的,登录后跳回那个 URL;否则才跳 /
  • defaultSuccessUrl("/", true):登录成功后强制跳 /,不跳回拦截前的 URL

什么时候用 true? 防开放重定向漏洞(Open Redirect):用户被诱导点击
/login?redirect=https://evil.com,登录后跳到恶意网站。设成 true 就只跳我们自己写的 URL,安全一些。

进阶路径(下一篇文章预告)

本 demo 是”内存版”,跑起来很方便,但生产肯定不行。下一步通常会走:

  1. 内存 → 数据库:UserDetailsService 改成从数据库查(JdbcUserDetailsManager / MyBatis-Plus 自己实现)
  2. Thymeleaf 前端鉴权:用 sec:authorize="hasRole('ADMIN')" 控制按钮是否显示
  3. 前后端分离:去掉 formLogin(),改成 JWT 认证 + 拦截器
  4. OAuth2 / 第三方登录:GitHub / 微信扫码登录

我会按这个顺序慢慢写,SpringSecurity实例2 应该是”数据库版”。

小结

整篇按”准备 → 核心实现 → 跑起来 → 补充知识点 → 进阶路径”五块过了一遍 Spring Security 的 RBAC 实例。
面试/复习重点看:

  • 认证流程:UsernamePasswordAuthenticationFilter → AuthenticationManager → UserDetailsService.loadUserByUsername()
  • 授权:URL 级别(requestMatchers + hasRole) + 方法级别(@PreAuthorize),两道防线
  • ROLE_ 前缀:hasRole 自动加,hasAuthority 不加,混用必踩坑
  • 多角色 vs RoleHierarchy:两种权限继承方式,场景不同
  • @EnableMethodSecurity:新写法,替代老的 @EnableGlobalMethodSecurity
  • CSRF:演示关、生产看场景(传统 SSR 开 / API 关)
  • 密码加密:必须用 BCrypt 之类的单向加密,不能存明文
