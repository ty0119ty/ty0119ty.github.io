BeanUtil.toBean 和 BeanUtils.copyProperties

我就说这两个怎么有点记混了,发现就是两个不同的工具包里面的功能

  1. BeanUtils.copyProperties
    来自 Apache Commons BeanUtils 或 Spring Framework 的
    org.springframework.beans.BeanUtils
  2. BeanUtil.toBean
    来自 Hutool工具库（一个国产Java工具包）import cn.hutool.core.bean.BeanUtil;

但是实际功能好像是一样的,这是我问了ai的
总结

  • 功能类似：两者都通过反射实现属性复制。
  • 差异点：Hutool的toBean更简洁，适合快速对象转换；Spring的copyProperties更适合Spring生态。
  • 性能：Hutool ≈ Spring > Apache Commons。

新的请求方式 Patch

PATCH：用于部分更新一个已存在的资源。你只提供需要改变的属性
PUT: 替换整个资源（完整更新）

在大多数情况下，使用 PATCH 进行部分更新的效率确实比使用 PUT 更高

  1. 网络传输效率 (最明显的优势)
    PATCH显著减少了请求体的大小，节省了上行带宽，降低了网络延迟，对于移动网络或慢速网络环境尤其重要
  2. 服务器处理效率
    PATCH减少了服务器端的CPU计算开销和处理时间
  3. 数据库操作效率
    PATCH可以带来更高效的数据库操作，减少数据库的负载和锁竞争
  4. 并发冲突概率
    PATCH降低了并发写冲突的风险，减少了处理冲突所需的逻辑

在现代API设计（RESTful API）中，推荐的实践是：

CREATE -> POST

READ -> GET

FULL UPDATE -> PUT

PARTIAL UPDATE -> PATCH

DELETE -> DELETE

BCrypt密码加密

MD5加密之后长度是32,而且同一个字符串加密之后都是相同的,
BCrypt加密之后是60,同一个字符串加密之后是不同的(添加了随机盐(随机字符串),更安全)

所以第五天最后是让我们自己完成用户管理模块,并将这个密码板块使用BCrypt加密进行操作,
不让用户密码在数据库里面明文显示
