前言

其实MySQL我主要还是背的黑马的那个,但是觉得有点少,
之后又去看了看二哥的,现在来也算是总结一下吧

之后要是有面试经历之类的,也可以写在这上面了

基础篇

什么是MySQL ?

mysql是一种开源的关系型数据库,利用二维表进行存储数据

char和varchar的区别 ?

  • 存储结构方面:char是定长字符,varchar是变长字符
  • 查询效率方面:一般情况下不会有太大的区别,但是当我们要考虑比较强大的性能要求时,
    我们可以将varchar变为char,比如UUID的36位字符就用char(36)

存储引擎

MySQL有哪些常见存储引擎 ?

MySQL的常见存储引擎有InnoDB,MyISAM和memory,
其中InnoDB是作为MySQL的默认存储引擎,原因:

  • InnoDB支持事务,但是其他两个不支持
  • InnoDB支持行级锁,锁的颗粒度更小,但是其他两个不支持
  • InnoDB还能通过日志实现崩溃恢复,保证了数据的一致性和持久性

但是在实际项目中,要是有其他需求,还是需要按需选择存储引擎

日志

MySQL的日志有哪些

MySQL的日志主要分为6类

  1. 错误日志(error log) – 用于问题诊断
  2. 慢查询日志(slow query log) – 用于SQL性能分析
  3. 一般查询日志(general log) – 记录所有SQL语句
  4. 二级制日志(binlog) – 用于主从复制和数据恢复
  5. 重做日志(redo log) – 保证事务的持久性
  6. 回滚日志(undo log) – 用于事务回滚和MVCC

为什么要两个阶段提交呢

保证redo log和binlog数据一致性,防止主从复制和事务状态不一致

redo log 的写入过程了解吗

InnoDB 会先将Redo Log 写入内存中的 Redo Log Buffer，
之后再以一定的频率刷入到磁盘的 Redo Log File 中

SQL优化

什么是慢SQL

MySQL中有一个long_query_time的参数,原则上执行时间超过这个参数的值,就是慢SQL

如何优化SQL

方法非常多,但是本质上都是一句话:

  • 尽可能少地扫描,尽快地返回结果

常见做法就是加索引,改写sql让它用上索引,比如使用索引覆盖,让联合索引遵循最左前缀原则,
还有别的就是优化查询条件、减少返回字段等,但是一般这俩在业务里面很少出现

explain平常有用过吗

经常用,explain 是 MySQL 提供的一个用于查看 SQL 执行计划的工具,
可以帮助我们分析查询语句的性能问题

索引

为什么索引就能提高效率

索引就像一本书的目录,能让 MySQL 快速定位数据,避免全表扫描,
它一般是 B+ 树结构,查找效率是 O(log n),比从头到尾扫一遍数据要快得多,
除了查得快,索引还能加速排序、分组、连接等操作
项目中最常见的做法就是通过 create index 为经常用作查询条件的字段建索引

能简单说一下索引的分类吗？

从功能上分类的话,有主键索引、唯一索引、全文索引;
从数据结构上分类的话,有 B+ 树索引、哈希索引;
从存储内容上分类的话,有聚簇索引、非聚簇索引。

  • 主键索引用于唯一标识表中的每条记录,其列值必须唯一且非空,创建主键时,MySQL 会自动生成对应的唯一索引

创建索引的时候有哪些需要注意的点

  • 选择合适的字段
  • 控制索引数量,不是越多越好,会占用存储空间
  • 联合索引的时候要遵循最左前缀原则

索引失效

  1. 违反最左前缀原则
  2. 给索引加上了函数或者参与运算
  3. 对索引进行了数据类型的转换
  4. 使用了like这样的模糊查询

为什么InnoDB要使用B+树作为索引

B+树是一种高效的多路平衡查找树,能有效的降低查询操作时的I/O次数,并且支持有序遍历和范围查询

  1. 数据只存在于叶子节点,非叶子节点只存索引
  2. 叶子节点之间形成双向链表,有利于范围查询
  3. 作为多路平衡二叉树,能存储的数据更多,降低树的高度,减少I/O次数

为什么用 B+ 树不用 B 树

  1. B 树的每个节点既存储键值,又存储数据和指针,导致单节点存储的键值数量较少
  2. B 树的范围查询需要通过中序遍历逐层回溯;而 B+ 树的叶子节点通过双向链表顺序连接,
    范围查询只需定位起始点后顺序遍历链表即可,没有回溯开销
  3. 因为B 树每个节点都会有值,那么查询效率是波动的,不稳定的;而B+树非常稳定,因为数据都在叶子节点

聚簇索引和非聚簇索引

  1. 聚簇索引存储了完整的数据行,默认情况下主键就是聚簇索引,用主键查询会非常的快
  2. 非聚簇索引的叶子节点存储的是主键值,查询的话是先查主键,再查主键里面的数据(这里就是再一次利用聚簇索引),会形成回表查询
  3. 不想回表查询,可以通过覆盖索引的方式,将要查询的字段也放在索引中

回表是什么

当使用非聚簇索引进行查询时,MySQL 需要先通过非聚簇索引找到主键值,
然后再根据主键值回到聚簇索引中查找完整数据行,这个过程称为回表

什么情况下会触发回表

  • 当查询字段不在非聚簇索引中时，必须回表到主键索引获取数据。
  • 查询字段包含非索引列（如 SELECT *），必然触发回表

联合索引了解吗

联合索引就是把多个字段放在一个索引里,但必须遵守“最左前缀”原则,
只有从第一个字段开始连续使用,索引才会生效;
联合索引会按字段顺序构建B+树,例如（age, name）索引会先按照 age 排序,
age 相同则按照 name 排序,若两者都相同则按主键排序,确保叶子节点无重复索引项。

创建(A,B,C)联合索引相当于同时创建了(A)、(A,B)和(A,B,C)三个索引

覆盖索引了解吗

覆盖索引指的是:查询所需的字段全部都在索引中,不需要回表,从索引页就能直接返回结果,
可以将高频查询的字段（如 WHERE 条件和 SELECT 列）组合为联合索引,实现覆盖索引

讲一下最左前缀原则

最左前缀原则指的是：MySQL 使用联合索引时,必须从最左边的字段开始匹配,才能命中索引
假设有一个联合索引 (A, B, C)，其生效条件如下：

查询条件 是否触发索引？ 说明
WHERE A = 1 ✅ 是 使用索引的第一列
WHERE A = 1 AND B = 2 ✅ 是 使用索引的前两列
WHERE A = 1 AND B = 2 AND C = 3 ✅ 是 使用索引的全部列
WHERE B = 2 ❌ 否 跳过左侧列 A，索引失效
WHERE B = 2 AND C = 3 ❌ 否 无左侧列，索引失效
WHERE A = 1 AND C = 3 ⚠️ 部分生效 仅用 A 列，C 列无法利用索引优化

锁

MySQL里面有哪几种锁

MySQL 中有多种类型的锁，可以从不同维度来分类

  • 按锁粒度划分的话,有表锁、行锁
  • 按照加锁机制划分的话,有乐观锁和悲观锁
  • 按照兼容性划分的话,有共享锁和排他锁

讲一下MySQL的行级锁

行锁是 InnoDB 存储引擎中最细粒度的锁,它锁定表中的一行记录,允许其他事务访问表中的其他行,
底层是通过给索引加锁实现的,这就意味着只有通过索引条件检索数据时,
InnoDB 才能使用行级锁,否则会退化为表锁

讲一下MySQL的乐观锁和悲观锁

悲观锁是一种”先上锁再操作”的保守策略,它假设数据被外界访问时必然会产生冲突,
因此在数据处理过程中全程加锁,保证同一时间只有一个线程可以访问数据

MySQL 中的行锁和表锁都是悲观锁

乐观锁会假设并发操作不会总发生冲突,属于小概率事件,因此不会在读取数据时加锁,
而是在提交更新时才检查数据是否被其他事务修改过

乐观锁并不是 MySQL 内置的锁机制,而是通过程序逻辑实现的,常见的实现方式有版本号机制和时间戳机制;
通过在表中增加 version 字段或者 timestamp 字段来实现

事务

讲一下MySQL事务的四大特性

  1. 原子性:保证事务中的操作要么全部执行、要么全部失败
  2. 一致性:保证数据从事务开始前的一个一致状态转移到结束后的另外一个一致状态
  3. 隔离性:保证并发事务之间互不干扰
  4. 持久性:保证事务提交后数据不会丢失

说一下事务的隔离级别

  • 读未提交:可能出现脏读
  • 读已提交:可能出现不可重复读
  • 可重复读:可能出现幻读(是MySQL默认的隔离机制)
  • 串行化:最安全,但是性能最差

说一下幻读呢

幻读是指在同一个事务中,多次执行相同的范围查询,结果却不同;
这种现象通常发生在其他事务在两次查询之间插入或删除了符合当前查询条件的数据

MVCC了解吗

MVCC 指的是多版本并发控制,每次修改数据时,都会生成一个新的版本,而不是直接在原有数据上进行修改;
并且每个事务只能看到在它开始之前已经提交的数据版本;
这样的话,读操作就不会阻塞写操作,写操作也不会阻塞读操作,从而避免加锁带来的性能损耗
