Java并发编程知识点-1

发表于2025-12-10|更新于2026-06-24|八股Java
|浏览量:

前言

并发编程作为企业级的一些基本能力,作为应届生的我,掌握部分即可,多线程的操作一般在普通项目中也用不到,
大型项目加上并发之后还有分布式,redis集群等等的进阶操作,我还是慢慢学吧

基础

说一下进程和线程的区别

进程相当于一个房子,线程相当于房子里面的房间,多个房间共享一个房子的资源

进程说简单点就是我们在电脑上启动的一个个应用;它是操作系统分配资源的最小单位
线程是进程中的独立执行单元;多个线程可以共享同一个进程的资源,
如内存,每个线程都有自己独立的栈和寄存器

说一下线程的几种创建方式

  1. 继承Thread类:
    • 重写父类的run()方法,并且调用start()方法启动
    • 缺点是不能多继承,如果已经继承了其他类,就无法继承Thread类
  2. 实现Runnable接口:
    • 重写run()方法,将实现类对象作为参数传递给Thread的构造方法,再调用start()方法启动
    • 这样避免了单继承的限制,并且更符合面向对象思想,降低耦合度
  3. 实现Callable接口:
    • 重写call()方法,然后创建 FutureTask 对象,参数为 Callable 实现类的对象;
      紧接着创建 Thread 对象,参数为 FutureTask 对象,最后调用 start() 方法启动线程
    • 优点是可以获取线程的执行结果
  4. 使用线程池(实际开发最常用):
    • 用 ExecutorService 配合 Executors 工厂,或直接用 ThreadPoolExecutor 构造
    • 优点是线程可复用,避免频繁创建/销毁线程的开销;更方便管理(统一调度、监控)
    • 详细用法见本文 # 线程池 一节

为什么不直接调用run(),而是用start()

调用start()会创建一个新的线程,并异步执行run()方法中的代码;
直接调用run()方法只是一个普通的同步方法调用,所有代码都在当前线程中执行,不会创建新线程;
没有新的线程创建,也就达不到多线程并发的目的

main (run方法调用的)
Thread-0 (start方法调用的)
可以中间加sleep自己看

线程有几种状态

6种;

new 代表线程被创建但未启动;
runnable 代表线程处于就绪或正在运行状态,由操作系统调度;
blocked 代表线程被阻塞,等待获取锁;
waiting 代表线程等待其他线程的通知或中断;
timed_waiting 代表线程会等待一段时间,超时后自动恢复;
terminated 代表线程执行完毕,生命周期结束

如何强制停止线程

  • 调用interrupt()方法,请求终止线程;
  • 在run()方法里面检查中断,如果捕获到了异常,就在run里面调用interrupt;

为什么不推荐 stop()/suspend()/resume()

  • stop():强制终止,会直接释放掉线程持有的所有锁——可能导致共享资源处于不一致状态(对象只被改了一半),不保证原子性。已废弃
  • suspend() / resume():挂起线程但不释放锁,容易造成死锁(其他线程等不到锁)。已废弃
  • 正确做法就是 interrupt() + 在 run() 里检查 Thread.currentThread().isInterrupted() 或捕获 InterruptedException

说一说 sleep 和 wait 之间的区别

sleep 会让当前线程休眠,不需要获取对象锁,属于 Thread 类的方法;
wait 会让获得对象锁的线程等待,要提前获得对象锁,属于 Object 类的方法;

详细解释一下:

  • 所属类不同:Thread和Object
  • 锁行为不同:
    • sleep期间要是有有锁,那么这个锁会一起sleep
    • wait期间有锁,那么这个锁会释放出去,给其他线程竞争
  • 使用条件不同:
    • sleep可以在任何地方调用
    • wait必须在同步代码块/同步方法种调用,因为前提是要当前线程必须有锁
  • 唤醒方式不同:
    • 调用 sleep 方法后,线程会进入 TIMED_WAITING 状态,即在指定的时间内暂停执行;
      当指定的时间结束后,线程会自动恢复到 RUNNABLE 状态,等待 CPU 调度再次执行
    • 调用 wait 方法后,线程会进入 WAITING 状态,直到有其他线程在同一对象上调用 notify 或 notifyAll 方法;
      线程才会从 WAITING 状态转变为 RUNNABLE 状态,准备再次获得 CPU 的执行权

怎么保证线程安全

为了保证线程安全,可以使用synchronized关键字对方法加锁,对代码块加锁;
线程在执行同步方法、同步代码块时,会获取类锁或者对象锁,其他线程就会阻塞并等待锁

  • 如果需要更细粒度的锁,可以使用 ReentrantLock 并发重入锁等
  • 如果需要保证变量的内存可见性,可以使用 volatile 关键字
  • 对于简单的原子变量操作,还可以使用 Atomic 原子类
  • 对于线程独立的数据,可以使用 ThreadLocal 来为每个线程提供专属的变量副本
  • 对于需要并发容器的地方,可以使用 ConcurrentHashMap、CopyOnWriteArrayList 等

ThreadLocal

ThreadLocal是什么

ThreadLocal 是一种用于实现线程局部变量的工具类;
它允许每个线程都拥有自己的独立副本,从而实现线程隔离;

使用步骤一般为:

  1. 创建ThreadLocal
  2. 设置ThreadLocal的值
  3. 获取ThreadLocal的值
  4. 删除ThreadLocal的值 优点:
  • 每个线程访问的变量副本都是独立的,避免了共享变量引起的线程安全问题
  • 由于 ThreadLocal 实现了变量的线程独占,使得变量不需要同步处理,因此能够避免资源竞争
  • ThreadLocal 可用于跨方法、跨类时传递上下文数据,不需要在方法间传递参数

ThreadLocal怎么实现

ThreadLocal 的实现原理是,每个线程维护一个 Map,key 为 ThreadLocal 对象,value 为想要实现线程隔离的对象

  1. 通过 ThreadLocal 的 set 方法将对象存入 Map 中
  2. 通过 ThreadLocal 的 get 方法从 Map 中取出对象
  3. Map 的大小由 ThreadLocal 对象的多少决定

ThreadLocal内存泄漏是怎么回事

ThreadLocalMap 的 Key 是弱引用,但 Value 是强引用
如果一个线程一直在运行并且 value 一直指向某个强引用对象,那么这个对象就不会被回收,从而导致内存泄漏

为什么 Key 设计成弱引用

如果 Key 是强引用,即使 localVariable = null,ThreadLocal 对象也不会被回收——因为 ThreadLocalMap 还强引用着它。
弱引用保证:只要 ThreadLocal 外部引用被置为 null,GC 时 Key 就能被回收,但 Value 仍然是强引用,所以仍然可能泄漏。

怎么避免内存泄漏(必做)

  1. 每次用完调用 remove(),显式清除 value
  2. 尽量把 ThreadLocal 声明为 static final,保证生命周期与类一致,避免频繁创建 ThreadLocal
  3. 在线程池场景下尤其重要——线程是复用的,ThreadLocal 不 remove 会一直带着

Java内存模型

谈一谈你对Java内存模型的理解

Java 内存模型是 Java 虚拟机规范中定义的一个抽象模型,用来描述多线程环境中共享变量的内存可见性

共享变量存储在主内存中,每个线程都有一个私有的本地内存,存储了共享变量的副本

  • 当一个线程更改了本地内存中共享变量的副本,它需要 JVM 刷新到主内存中,以确保其他线程可以看到这些更改
  • 当一个线程需要读取共享变量时,它一般会从本地内存中读取;
    如果本地内存中的副本是过时的,JVM 会将主内存中的共享变量最新值刷新到本地内存中

volatile了解吗

了解

  • 保证可见性:线程修改 volatile 变量后,其他线程能够立即看到最新值
  • 防止指令重排:volatile 变量的写入不会被重排序到它之前的代码

volatile 不保证原子性

count++ 实际是 3 步:读、改、写。volatile 只保证”读到的最新”,不保证”读-改-写”整体是原子的。
想要原子自增,要用 AtomicInteger.incrementAndGet(),基于 CAS。

volatile vs synchronized

维度 volatile synchronized
作用 修饰变量 修饰方法/代码块
原子性 ❌ 不保证 ✅ 保证
可见性 ✅ 保证 ✅ 保证
阻塞 不会阻塞线程 会阻塞其他线程
适用 单写多读、状态标志 复合操作、临界区

锁

synchronized锁升级了解吗

注意:JDK 15 之后偏向锁已被默认废弃,新版本 JVM 在没有竞争时不再走偏向锁,
直接是无锁 → 轻量级锁 → 重量级锁三段式。
下面讲的是”完整 4 级升级”,了解历史即可。

会先从无锁升级为偏向锁,再升级为轻量级锁,最后升级为重量级锁

  1. 当线程第一次获取锁时,会进入偏向模式,锁升级为偏向锁
  2. 当多个线程在不同时段获取同一把锁,即不存在锁竞争的情况时,JVM 会采用轻量级锁来避免线程阻塞,
    轻量级锁通过自旋来等待锁释放
  3. 如果自旋超过一定的次数,或者一个线程持有锁,一个自旋,
    又有第三个线程进入 synchronized 加锁的代码时,轻量级锁就会升级为重量级锁

synchronized和ReentrantLock的区别了解吗

  • synchronized 由 JVM 内部的 Monitor 机制实现,ReentrantLock基于 AQS 实现;
  • synchronized 可以自动加锁和解锁,ReentrantLock 需要手动 lock() 和 unlock()

完整对比表

维度 synchronized ReentrantLock
实现 JVM 内置(Monitor) JDK 代码(AQS)
加锁/解锁 自动 必须手动 lock() / unlock()(一般 try-finally)
公平性 非公平 可选(构造传 true 公平,默认非公平)
可中断 不支持 lockInterruptibly()
尝试获取锁 不支持 tryLock()(返回布尔,可超时)
条件变量 只有 wait/notify Condition,可以多个等待队列
性能 JDK 6 之后优化,差距不大 高竞争场景略优

实际开发:简单场景用 synchronized,需要公平锁/可中断/超时/多条件变量时用 ReentrantLock。

讲一下悲观锁和乐观锁

悲观锁认为每次访问共享资源时都会发生冲突,所在在操作前一定要先加锁,防止其他线程修改数据

乐观锁认为冲突不会总是发生,所以在操作前不加锁,而是在更新数据时检查是否有其他线程修改了数据;
如果发现数据被修改了,就会重试

乐观锁的两种典型实现

1. 版本号机制(数据库常用)

如果返回 affected rows = 0,说明版本号已经被别人改过,需要重试。

2. CAS(Compare And Swap)(Java 并发包用)

CAS 逻辑:比较内存值 == 旧值? 是 → 改成新值; 否 → 重试 / 放弃

  • 优点:无锁,性能高
  • 缺点:ABA 问题(值从 A→B→A,看似没变,实际变了)。解决方案:加版本号(AtomicStampedReference)

典型例子

  • AtomicInteger / AtomicLong / AtomicReference / AtomicStampedReference
  • ConcurrentHashMap(1.8 之后)用 CAS + synchronized 替代 1.7 的分段锁
  • 数据库乐观锁:UPDATE ... WHERE version = ?

并发工具类

JDK 在 java.util.concurrent 包下提供了一组并发工具,常被问到的有 3 个:

CountDownLatch(倒计时门闩)

让一个或多个线程等待其他线程完成一组操作后才能继续。
类比:运动会跑步——发令枪响,各跑道选手起跑;主线程等所有选手跑完才公布成绩。

CyclicBarrier(循环栅栏)

让一组线程互相等待到同一个屏障点,然后大家一起继续。
和 CountDownLatch 的区别:

  • CountDownLatch 是”等别人干完”,一次性(计数到 0 之后就不能再用了)
  • CyclicBarrier 是”大家约好在某个点集合”,可循环使用(reset())

Semaphore(信号量)

控制同时访问某个资源的线程数。常用于限流。

三个工具对比表

工具 用途 关键特点
CountDownLatch 等 N 个任务完成 一次性、计数器只减不增
CyclicBarrier N 个线程互相等到屏障点 可复用、大家一起继续
Semaphore 限流(控制并发数) 许可证机制,acquire/release

线程池

线程池是面试最常被问到的并发话题。核心是:预先创建一批线程,反复利用,避免频繁创建/销毁。

为什么要用线程池

  • 降低资源消耗:线程复用,避免反复创建/销毁线程的开销
  • 提高响应速度:任务来时不用等新线程创建,直接拿现成的
  • 便于管理:可以统一监控、调优、拒绝策略

ThreadPoolExecutor 的 7 个参数(面试必背)

4 种拒绝策略

策略 行为
AbortPolicy(默认) 抛异常 RejectedExecutionException,最常用
CallerRunsPolicy 让提交任务的线程自己跑这个任务(慢但不会丢任务)
DiscardPolicy 静默丢弃,不抛异常(慎用,任务丢了都不知道)
DiscardOldestPolicy 丢弃队列里最老的一个任务,然后重新提交当前任务

Executors 工厂(不推荐在生产用)

  • newFixedThreadPool(n):固定大小,LinkedBlockingQueue(无界,可能 OOM)
  • newCachedThreadPool():可无限扩展,SynchronousQueue(短任务多时线程数爆掉)
  • newSingleThreadExecutor():单线程,串行执行
  • newScheduledThreadPool(n):定时/周期任务

    生产推荐直接用 ThreadPoolExecutor 显式传参,因为 Executors 的几个工厂默认队列/线程数有坑,容易在压力下出问题。

线程池执行流程(被问最多的图)

submit 和 execute 的区别

  • execute(Runnable):无返回值,异常会直接抛出
  • submit(Runnable/Callable):返回 Future,可以用 future.get() 取结果 + 异常被封装在 Future 里
