Java集合框架知识点-1

发表于2025-12-10|更新于2026-06-23|八股Java
|浏览量:

前言

集合框架这部分就有点深入了解一些数据结构了,比如list呀,map呀之类的,
这是作为一名基础开发人员所应该具备的基本开发技能

说一说有哪些常见的集合框架

  1. Collection:
    1. List(有序可重复集合):封装了常见的ArrayList(动态数组),LinkedList(链表)
    2. Set(无序不可重复集合):有HashSet和TreeSet
    3. Queue(队列):ArrayDeque(双端队列),PriorityQueue(优先级队列)
  2. Map:
    1. 键值对集合,有常见的HashMap,TreeMap等

一些常见的工具类,都在java.util包里面:

  • Collections：提供了一些对集合进行排序、二分查找、同步的静态方法
  • Arrays：提供了一些对数组进行排序、打印、和 List 进行转换的静态方法

List

讲一下ArrayList和LinkedList的区别

ArrayList 是基于数组实现的,LinkedList 是基于链表实现的;
多数情况下,ArrayList 更利于查找,LinkedList 更利于增删,
这是由他们的结构体现的,数组可以直接通过下标进行数据查找,而链表则需要遍历查找,
但是插入元素的话,插入数组中间则需要移动数组,而链表只需要修改前后节点就可以实现了

ArrayList 扩容机制

  • 初始容量:10(空构造时其实是 0,第一次 add 才变成 10)
  • 扩容方式:1.5 倍(int newCapacity = oldCapacity + (oldCapacity >> 1))
  • 底层是 Arrays.copyOf(),扩容时新建数组 + System.arraycopy() 复制
  • 所以频繁增删前,可以 new ArrayList<>(expectedSize) 预设容量,减少扩容次数

Fail-Fast(快速失败)

  • ArrayList 迭代时,如果其他线程结构性修改(add/remove)了列表,会立刻抛 ConcurrentModificationException
  • 原理是 modCount 字段:每次结构修改 ++modCount,迭代器创建时记录 expectedModCount,迭代时发现不一致就抛异常
  • 单线程下也要小心:在 forEach 里调用 list.add/remove 就会触发

ListIterator(双向迭代)

  • 比 Iterator 多几个方法:hasPrevious() / previous() / add() / set()
  • 可以在迭代过程中增删元素(通过 iterator 自己的方法,不会触发 Fail-Fast)

Map

说一下HashMap的底层数据结构

jdk8中,HashMap的底层结构为数组+链表+红黑树

数组用来存储键值对,每个键值对可以通过索引直接拿到,索引是通过对键的哈希值进行进一步的 hash() 处理得到的

当多个键经过哈希处理后得到相同的索引时,需要通过链表来解决哈希冲突—-将具有相同索引的键值对通过链表存储起来

不过,链表过长时,查询效率会比较低,于是当链表的长度超过 8 时(且数组的长度大于 64),链表就会转换为红黑树;
红黑树的查询效率是 O(logn),比链表的 O(n) 要快

HashMap的put流程

整体流程是:计算 hash → 寻址 → 判断桶位 → 处理冲突 → 插入 → 检查扩容

简化版:

关键点:**hash() 用了扰动函数 (h = key.hashCode()) ^ (h >>> 16)**——高低位混合,降低冲突。

HashMap 的扩容机制

  • 初始容量:16(必须是 2 的幂,因为寻址用 (n-1) & hash 比取模快)
  • 负载因子:0.75(空间/时间平衡点,数学上最优近似)
  • 扩容时机:size > threshold 时,threshold = 容量 × 负载因子
  • 扩容方式:2 倍扩容,重新 rehash,把元素重新分配到新桶
  • 树化阈值:链表长度 ≥ 8 且数组长度 ≥ 64 才转红黑树(否则先扩容)
  • 链表化阈值:红黑树节点数 ≤ 6 时退化成链表

HashMap 1.7 vs 1.8

维度 1.7 1.8
底层结构 数组 + 链表 数组 + 链表 + 红黑树
链表插入方式 头插法 尾插法
扩容时 头插法 + 链表反转 → 多线程下可能死循环 尾插法 + 红黑树,死循环问题解决
hash 扰动 4 次位运算 + 5 次异或 1 次位运算 + 1 次异或

头插法在多线程并发扩容时,会形成链表环,导致 get 死循环——1.8 改尾插 + 红黑树就是为了修这个。

HashMap 为什么线程不安全 / 怎么解决

  • 不安全点:modCount 非原子、++size 非原子、并发扩容可能死循环(1.7)
  • 解决:
    • Collections.synchronizedMap(map):整个 map 加锁,性能差
    • ConcurrentHashMap:1.7 用分段锁 Segment(16 段),1.8 改用 CAS + synchronized(锁桶头节点),并发度更高

HashMap vs Hashtable vs ConcurrentHashMap 对比

维度 HashMap Hashtable ConcurrentHashMap
线程安全 ❌ ✅(方法全 synchronized) ✅
性能 最高 最差(整表锁) 高(锁粒度细)
null key/value 允许 不允许 不允许
迭代器 Fail-Fast Enumerator(非 Fail-Fast) Fail-Fast(弱一致)
适用 单线程 基本不用了 多线程并发场景

Fail-Fast:迭代过程中如果其他线程修改了 map,会立刻抛 ConcurrentModificationException。

HashSet 怎么去重

  • 底层其实就是 HashMap,add 一个元素时,value 永远是同一个 PRESENT 对象
  • 去重靠的是 hashCode() + equals():
    • 先比较 hash(快速定位桶位)
    • 再 equals 比较内容(确认是否同一个对象)
  • 重写 equals 时必须重写 hashCode,否则可能”equals 相等但 hash 不同”——同一个对象会被放进不同的桶,导致重复

集合选型指南(面试常考)

  • 查多改少 → ArrayList(底层数组,按下标 O(1))
  • 频繁增删 → LinkedList(注意:实际中 ArrayList 也不慢,因为 CPU 缓存友好;真正高频插入删除才考虑 LinkedList)
  • 去重 → HashSet;有序去重 → LinkedHashSet;排序去重 → TreeSet
  • key-value 查询 → HashMap;需要排序 → TreeMap;保留插入顺序 → LinkedHashMap;线程安全 → ConcurrentHashMap

小结

整篇按”List → Map”两块过了一遍 Java 集合框架的高频八股点。
面试前重点看:

  • ArrayList vs LinkedList:各自底层 + 适用场景 + ArrayList 1.5 倍扩容
  • Fail-Fast:modCount 机制,迭代时不能结构性修改
  • HashMap put 流程:hash → 寻址 → 桶位判断 → 链表/树 → 扩容
  • HashMap 扩容:16 初始、0.75 负载因子、2 倍扩容、树化阈值 8、链表化阈值 6
  • HashMap 1.7 vs 1.8:头插 → 尾插、链表 → 红黑树、死循环问题修复
  • HashMap vs Hashtable vs ConcurrentHashMap:null 允许、线程安全、锁粒度
  • HashSet 底层 = HashMap:hashCode + equals 共同决定去重
  • 选型指南:ArrayList/HashMap 用得最多,LinkedList 慎用,TreeXxx 留给真要排序的场景
