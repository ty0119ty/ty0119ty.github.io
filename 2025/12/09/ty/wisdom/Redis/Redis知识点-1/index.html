Redis知识点-1

发表于2025-12-09|更新于2026-06-24|八股Redis
|浏览量:

前言

作为后端开发中数据操作最主要的中间件之一,这也是基本开发需求所需要具备的能力

基础

什么是Redis说一下

Redis是一种基于键值对的NoSQL数据库(非关系型数据库);
它的主要特点就是把数据放在内存里面,提高读写速度,面对一些对性能要求很高的场景,
比如:缓存热点数据,频繁接口调用,都会用到Redis

Redis的数据类型

  1. String(字符串):可以用来存储token,验证码等等,常见于单个对象
  2. List(列表):可以用来缓存消息队列,任务列表等
  3. Hash(哈希):键值对集合,适合用来存储对象,如商品信息,用户信息等等
  4. Set(集合):查询效率很高,能达到O(1),主要用于去重,标签,共同好友等
  5. ZSet(有序集合):支持范围查询,常用于排行榜,优先级队列等

数据类型底层结构(高频考点)

类型 底层结构 编码(redisObject.ptr 指向)
String SDS(简单动态字符串) embstr(≤44字节) / raw
List quicklist(3.2+) 由多个 listpack 通过双向指针串起来
Hash listpack(7.0+) 之前是 hashtable + ziplist
Set intset(全整数且少) / hashtable 按数据自动转换
ZSet skiplist + hashtable listpack(元素少且分数小) / skiplist

SDS 比 C 字符串的优势:O(1) 取长度、不溢出、二进制安全、预分配冗余空间减少分配次数。

为什么Redis快

  1. 最简单直观的,数据存在内存里面,I/O操作相比于去磁盘里面,快了几个数量级
  2. Redis 采用了基于 I/O 多路复用技术的事件驱动模型来处理客户端请求和执行 Redis 命令
  3. Redis 对底层数据结构做了极致的优化,比如String的底层动态字符串支持动态扩容,预分配冗余空间,
    能够减少内存碎片和内存分配的开销
  4. C 语言实现,无 GC 开销,接近底层

Redis 是单线程吗

  • Redis 6.0 之前:完全是单线程(命令执行 + I/O 都在一个线程)
  • Redis 6.0+:I/O 多线程(网络读写多线程),但命令执行仍然是单线程
  • 也就是说:任意命令的执行是原子的,不会被打断——这是 Redis 既快又能用 INCR 做计数器的原因
  • 为什么不全部多线程?CPU 瓶颈不是主要矛盾,内存和 I/O 才是,多线程反而带来锁开销

持久化

Redis的持久化方式

有两种方式:

  1. RDB:通过创建时间点快照来实现持久化
  2. AOF:通过记录每个写操作命令来实现持久化

这两种方式可以单独使用,也可以同时使用;
这样就可以保证 Redis 服务器在重启后不丢失数据,通过 RDB 和 AOF 文件来恢复内存中原有的数据

详细说一下RDB

RDB 持久化机制可以在指定的时间间隔内将 Redis 某一时刻的数据保存到磁盘上的 RDB 文件中,
当 Redis 重启时,可以通过加载这个 RDB 文件来恢复数据

RDB 触发机制

  • 手动:save(同步阻塞,会卡住) / bgsave(后台 fork 子进程,推荐)
  • 自动:save 900 1 表示 900 秒内至少 1 次修改就触发
  • 优:文件小、恢复快;缺:可能丢数据(最后一次快照之后到宕机之间的数据)

详细说一下AOF

AOF 通过记录每个写操作命令,并将其追加到 AOF 文件来实现持久化,
Redis 服务器宕机后可以通过重新执行这些命令来恢复数据

AOF 三种 fsync 策略

策略 行为 性能 安全性
always 每条命令都 fsync 最差 不丢数据
everysec(默认) 每秒 fsync 一次 折中 最多丢 1 秒数据
no 由 OS 控制 最好 宕机可能丢很多数据

AOF 重写:AOF 文件会越写越大,Redis 提供 bgrewriteaof 重新生成一份”等价但最小”的 AOF,
原理是遍历内存中的数据,反向生成命令(而不是读旧 AOF 文件)。

RDB vs AOF 对比

维度 RDB AOF
文件大小 小(快照) 大(每条命令)
恢复速度 快 慢(逐条执行)
数据安全性 可能丢 最多丢 1 秒
适用 备份 / 容忍丢数据 不能丢数据
生产建议 两者一起用(混合持久化) 两者一起用

Redis4.0的混合持久化了解吗

混合持久化结合了 RDB 和 AOF 两种方式的优点,解决了它们各自的不足

RDB恢复数据快,但是快照之间的时间间隔会导致数据丢失
AOF记录了每一次的操作,保证数据的持久性,但是一步一步恢复数据特别慢

原理就是先将RDB里面的数据加进去,再去找AOF中留存的数据;

缓存设计

缓存击穿,穿透,雪崩

缓存击穿

缓存击穿是指某个热点数据缓存过期时,大量请求就会穿透缓存直接访问数据库,导致数据库瞬间承受的压力巨大
解决方案:

  1. 互斥锁,当缓存失效的时候,第一个访问的线程先获取锁并负责重建缓存,其余线程等待/重试
  2. 永不过期策略,缓存项本身不设置过期时间,也就是永不过期,但在缓存值中维护一个逻辑过期时间;
    当缓存逻辑上过期时,返回旧值的同时,异步启动一个线程去更新缓存

缓存穿透

缓存穿透是指查询的数据在缓存中没有命中,因为数据压根不存在,所以请求会直接落到数据库上;
原因可能是自身的业务代码有问题,或者是恶意攻击造成的,比如爬虫
解决方案:

  1. 布隆过滤器,是一种空间效率很高的数据结构,可以用来判断一个元素是否在集合中
  2. 缓存空值,对于不存在的数据,我们将空值写入缓存,并设置一个合理的过期时间;
    这样下次相同的查询就能直接从缓存返回,而不再访问数据库

缓存雪崩

缓存雪崩是指在某一时间段,大量缓存同时失效或者缓存服务突然宕机了;
导致大量请求直接涌向数据库,导致数据库压力剧增,甚至引发系统崩溃的现象
成因及其解决方案:

  1. 大量缓存同时过期–>添加随机过期时间
  2. 缓存服务崩溃–>使用高可用的缓存集群
  3. 缓存服务正常但并发请求量超过了缓存服务的承载能力–>采用限流和降级措施

能说一下布隆过滤器吗

布隆过滤器是一种空间效率极高的概率性数据结构,用于快速判断一个元素是否在一个集合中;
它的特点是能够以极小的内存消耗,判断一个元素“一定不在集合中”或“可能在集合中”,
常用来解决 Redis 缓存穿透的问题

布隆过滤器可能会错误地认为某个元素在集合中,而元素实际上并不在集合中
但是,布隆过滤器认为某个元素不存在于集合中,那么它一定不存在

内存管理

Redis 内存淘汰策略(面试最高频)

Redis 内存不够用时(达到 maxmemory),按以下策略淘汰 key。8 种策略分两类:

不淘汰(只对带过期时间的 key 操作)

策略 行为
volatile-lru 在设置了过期时间的 key 里,淘汰最近最少使用的
volatile-lfu 在设置了过期时间的 key 里,淘汰使用频率最低的(Redis 4.0+)
volatile-ttl 在设置了过期时间的 key 里,淘汰剩余 TTL 最短的
volatile-random 在设置了过期时间的 key 里,随机淘汰

全键淘汰(所有 key 都可能被淘汰)

策略 行为
allkeys-lru 所有 key 中,LRU 淘汰(最常用)
allkeys-lfu 所有 key 中,LFU 淘汰(更精准)
allkeys-random 所有 key 中,随机淘汰
noeviction(默认) 不淘汰,写命令直接返回错误(读命令正常)

生产建议:allkeys-lru(缓存场景,不知道未来访问模式)或 volatile-lru(部分 key 必须保留,如 session)。

过期键删除策略

Redis 用两种策略配合清理过期 key:

  • 惰性删除:访问 key 时才检查是否过期,过期就删除
  • 定期删除:每隔 100ms 随机抽取一批 key 检查过期

为什么不立即删除:立即删除要 CPU 实时扫所有 key,代价太大;
惰性删除 + 定期删除是 CPU 和内存的折中。但已经过期的 key 不一定会马上被删,这就是为什么 TTL 命令可能返回 -2。

如何保证数据库和缓存数据一致性

读操作写入redis,增删改操作删除redis

读取时先查 Redis,未命中再查 MySQL,同时为缓存设置一个合理的过期时间;
更新时先更新 MySQL,再删除 Redis
这种方式简单有效,适用于读多写少的场景

Cache Aside 模式(最常用)

读:

  1. 读 Redis,命中 → 返回
  2. 未命中 → 读 MySQL → 写回 Redis → 返回

写:

  1. 先写 MySQL
  2. 再删 Redis(不是更新,是因为更新可能并发写覆盖)

    为什么是”删”不是”改”?避免脏读:如果两个写请求并发,都”先改缓存再改 DB”,
    慢的请求可能用旧值覆盖快的请求的缓存。删缓存只删一次,下次读时自然从 DB 加载最新值。

延迟双删(进一步降低脏读概率)

为什么第二次还要删? 上面的步骤 2 删除后,可能正好有个并发”读 MySQL → 写 Redis”把旧值又写进缓存。
延迟双删就是用 sleep 覆盖掉这个”读慢、写回旧值”的时间窗,再删一次兜底。

强一致性 vs 最终一致性

  • 强一致:Redis 和 MySQL 任何时刻都一样——基本做不到,网络/时钟都有误差
  • 最终一致(现实选择):容忍短暂不一致(几秒~几分钟后一致)
  • 如果业务要求”下单立即看到库存”,不要用 Redis;Redis 适合”展示性数据”(商品列表、用户信息)

Redis应用

Redis支持事务吗

Redis支持简单事务,可以将 multi、exec、discard 和 watch 命令打包,然后一次性的按顺序执行

  • 基本流程是用 multi 开启事务,然后执行一系列命令,最后用 exec 提交;
    这些命令会被放入队列,在 exec 时批量执行
  • discard 命令用于取消一个事务,它会清空事务队列并退出事务状态
  • watch 命令用于监视一个或者多个 key,如果这个 key 在事务执行之前 被其他命令改动,那么事务将会被打断
  • 但 Redis 的事务与 MySQL 的有很大不同,它并不支持回滚,也不支持隔离级别

为什么 Redis 事务没有回滚

  • MySQL 事务回滚靠 undo log,代价大(需要记录反向操作)
  • Redis 追求简单和性能,命令失败只记录,事务继续执行——认为程序应该提前检查
  • 所以 Redis 事务只能保证”批量执行”和”不被打断”,不能保证”全部成功”

Lua 脚本(更常用的”原子操作”)

  • 用 EVAL 或 EVALSHA 执行
  • 整个 Lua 脚本作为一个命令执行,原子
  • 比 MULTI/EXEC 更强大,可以做条件判断——生产里”扣库存””分布式锁”基本都用 Lua

集群方案

三种部署模式对比

模式 数据分布 高可用 容量 适用
主从复制 全量复制到从节点 手动切换 受单机限制 读写分离、备份
哨兵(Sentinel) 全量复制 自动故障转移 受单机限制 主从 + 自动切换
Cluster 分片(16384 个槽) 内置高可用 可水平扩展 大数据量、高并发

几个关键点

  • 主从复制:从节点只能读,主节点挂了需要手动切换(或者用哨兵自动切)
  • 哨兵:本质是一组监控进程,至少 3 个节点,投票选举新主节点
  • Cluster 哈希槽:16384 个槽位分布在多个主节点,key 用 CRC16(key) % 16384 算槽位
    • 优点:水平扩展,数据分片
    • 缺点:多 key 操作(MSET / 事务)只能在一个槽——可以用 hashtag {} 强制同槽,比如 user:{123}:name

小结

整篇按”基础 → 持久化 → 缓存设计 → 内存管理 → Redis 应用 → 集群方案”六块过了一遍 Redis 高频八股点。
面试前重点看:

  • 5 种数据类型 + 底层结构(String→SDS、List→quicklist、Hash→listpack、Set→intset/hashtable、ZSet→skiplist+hashtable)
  • Redis 单线程模型(6.0+ I/O 多线程但命令执行仍单线程 → 任意命令原子)
  • RDB vs AOF + 混合持久化(RDB 恢复快但可能丢,AOF 默认 everysec)
  • 缓存三件套:击穿(互斥锁/永不过期)/ 穿透(布隆过滤器/缓存空值)/ 雪崩(随机过期/集群/限流)
  • 数据库和缓存一致性:Cache Aside + 延迟双删(强一致做不到,选最终一致)
  • 8 种内存淘汰策略 + 过期键删除(惰性 + 定期)
  • 三种集群部署:主从 / 哨兵 / Cluster(16384 槽)
  • Redis 事务 vs MySQL 事务 + Lua 脚本(生产中比事务更常用)
