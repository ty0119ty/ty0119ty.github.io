简历拷打篇-AI面-1

发表于2025-12-15|更新于2026-02-05|简历
|浏览量:

前言

虽然自己在背八股,但是和真实的面试经历相比,还是有所差距或是实际的区别;
那现在就用AI来帮我面试一下,看一下会有哪些问题是还需要去实际掌握的;

本次选用的AI为–ChatGPT

面试拷打ChatGPT篇

提示词:

现在你是面试官,请你针对性的向我的简历提问,主要围绕实习经历,
第一个项目经历,由浅入深,循序渐进,最好有引导式提问,我回答之后,
先进行点评,在给出可以优化的回答或者给出相对完美的回答

基础拷打

第一问

你在乐鱼健康科技这段实习中,主要负责了哪些工作？
如果让我用一句话概括你的实习价值，你希望我怎么记住你？

  • 答:
    • 在乐鱼健康科技的实习中，我主要负责后端核心接口的性能优化和权限体系的改造。
    • 一方面我通过 Redis 缓存设计和 SQL 索引优化，对高频接口进行了重构，使核心接口响应时间从 500ms 降到 50ms 左右；
    • 另一方面基于 RBAC 模型整合 Spring Security，实现了多角色、多模块的权限隔离，支撑了平台在养老机构场景下的多人员使用。
    • 如果用一句话概括，我在实习中更多承担的是把系统跑稳、跑快的角色，并且这些优化都已经在实际业务中上线使用。

第二问(区分水平)

你刚才提到 接口性能从 500ms 优化到 50ms，
这个优化是一次性完成的，还是逐步定位问题做的？

能不能简单说一下：
你是如何定位到性能瓶颈的？

  • 答:
    • 是逐步定位完成的。
    • 首先我通过 慢日志和接口耗时统计，定位到几个访问频率高、响应时间明显偏长的核心接口；
    • 针对这些接口，我使用 EXPLAIN 分析 SQL 执行计划，重点关注 type、rows 和是否命中索引，当发现 rows 扫描行数偏大或出现 ALL 全表扫描时，就针对性地进行索引优化或 SQL 改写；
    • 在 SQL 优化到一定程度后，对于读多写少的接口，我进一步引入 Redis 缓存，将结果缓存起来，最终把接口响应时间稳定控制在 50ms 左右。

第三问(开始卡人)

你刚刚提到 Redis 缓存，
那我想追问一个比较关键的问题：
你是如何保证 Redis 缓存和 MySQL 数据一致性的？

为什么你选择的是
“先写 MySQL，再删 Redis” 这个策略？

  • 答:
    • 我们采用的是 “先写 MySQL，再删 Redis” 的策略，来保证缓存与数据库的最终一致性。
    • 原因是如果先操作缓存再更新数据库，在并发读写场景下，可能会出现旧数据被重新写回缓存的问题。
    • 而先更新数据库，再删除缓存，即使在极端并发场景下出现短暂的不一致，后续读请求在缓存失效后，仍然会从数据库加载到最新数据并回填缓存，从而保证最终一致。
    • 同时在实现上，我们也会对删除缓存操作进行重试或异步补偿，避免因为删除失败导致脏数据长期存在。

第四问(重点卡点)

你刚刚说到了 并发场景，那我继续追问：
在高并发下，
“先写 MySQL，再删 Redis” 一定不会出现不一致吗？

如果不能完全避免，你们是如何接受或缓解这个问题的？

  • 答:
    • 这一块在实际项目中我们主要采用的是
      “先写 MySQL，再删 Redis” 这一种方案，
      对于极端并发下的短暂不一致问题，目前还没有深入到复杂方案。
    • 但我自己是知道，在高并发场景下这种策略并不能做到强一致，只能保证最终一致性，
    • 实际生产中一般会通过重试、延时双删或者异步补偿等方式来进一步降低风险。
    • 这一部分我目前更多是停留在理解阶段，后续如果接触到更高并发的业务，我也会重点去补这一块。

细节问题

第五问(权限模型RBAC)

你在实习中提到 基于 RBAC 模型集成 Spring Security，实现多角色权限控制。
先不谈代码细节，
你能先说一下你们系统里一共有哪几类角色？

以及不同角色之间权限差异是如何体现的吗？

  • 答:
    • 我们系统中是 典型的 RBAC 权限模型，
      按照养老机构的实际业务划分了多种角色，比如 护士、财务主管、主任、系统管理员 等。
    • 权限并不是直接绑在用户上的，而是按功能模块进行拆分，再分配给角色。
      比如护士角色主要拥有 老人信息查看、房间分布查看 等只读权限；
      而主任角色则额外拥有 服务项目和费用配置 等管理权限。
    • 通过这种方式，当人员岗位发生变动时，只需要调整角色绑定关系，而不需要逐一修改用户权限，维护成本会比较低。

第六问(考 Spring Security)

那我继续追问一下实现层面的问题：

在 Spring Security 里，你们是如何把“角色/权限”
和具体接口（URL 或方法）绑定起来的？

比如护士不能访问某个接口，这个限制是在哪里做的？

  • 答:
    • 在 Spring Security 中，我们是通过 角色和权限标识 来做访问控制的，并不是简单的等级判断。
    • 在配置层，我们会将接口 URL 或方法与具体的 角色（ROLE）或权限（Authority） 进行绑定，
      比如通过配置哪些接口需要 ROLE_NURSE，哪些需要 ROLE_DIRECTOR。
    • 当用户登录后，系统会将该用户所拥有的角色信息加载到 SecurityContext 中，
      Spring Security 会在请求经过过滤器链时，自动判断当前用户是否具备访问该接口的权限。

第七问(高频)

那我再具体一点问你：

用户登录之后，
Spring Security 是在什么地方把“用户角色信息”
放到当前请求上下文中的？

也就是说，后续请求是怎么知道
“当前用户是谁、有什么权限”的？

  • 答:
    • 在我们系统中，用户登录成功后会生成一个 JWT 返回给前端，
      前端在后续请求中通过请求头携带这个 Token。
    • 请求进入后端时，会先经过 Spring Security 的过滤器链，
      在自定义的认证过滤器中解析 JWT，获取用户的身份信息和角色权限，
      并封装成一个 Authentication 对象，
      然后存入当前线程的 SecurityContext 中。
    • 后续在接口访问阶段，Spring Security 就是通过
      SecurityContext 中的 Authentication 信息，
      来判断当前用户是否具备访问该接口的权限。

第八问(压力)

你简历里写到：

接口响应时间 500ms 优化到 50ms
并发用户 1000+
缓存命中率 85%

这些数据是你真实测出来的吗？
你是怎么评估和验证这些指标的？

  • 答:
    • 这些数据里，接口响应时间的优化是我自己参与测试和验证的。
      我在优化前后使用压测工具对核心接口进行了对比测试，
      能明显看到响应时间从大约 500ms 降到 50ms 左右。
    • 并发用户数和缓存命中率这类整体系统指标，
      是由团队统一做压测时统计出来的，
      我不是主导测试的人，但在过程中我参与了相关模块的优化，
      并且在测试后查看过 Redis 监控和服务日志，
      确认缓存命中率和并发提升与我负责的改动是对应得上的。

第九问(终极压力)

如果让你现在回头看这段实习，
你觉得自己做得最不足的一点是什么？
如果再给你一次机会，你会重点补哪一块？

  • 答:
    • 回头看这段实习，我觉得自己做得最不足的一点是，
      在遇到问题时有时候会比较着急，
      第一反应是去问组长或同事，而不是先完整地自己拆解一遍问题。
    • 后来我发现，其实不少问题通过看日志、定位报错、逐步排查，
      自己是可以解决的，只是当时经验不足，心态上容易慌。
    • 如果再给我一次机会，我会更有意识地
      先独立分析问题、形成自己的判断，再带着结论去请教别人，
      这样既能提高效率，也能更快沉淀自己的技术能力。

项目–宠物交流社

第一问(判断项目真假)

你这个宠物交流社项目，
你觉得哪一块最难、最容易出问题？为什么？

  • 答:
    • 我觉得最难的新技术的应用吧,之前的实习和项目中都没有引入过消息队列MQ这个中间件,
      还有就是比如点赞评论这些消息的投递,会伴随着其他的比如用户活跃度的增加,
      一个事件串联的其他的很多事件的变化,就有点绕不过来了

第二问(追杀)

那我换个角度问你：

你为什么要用 RabbitMQ？
不用 MQ，直接在点赞接口里同步处理这些逻辑行不行？

  • 答:
    • 在这个项目里之所以使用 RabbitMQ，
      主要是因为像 点赞、评论、收藏 这种行为，
      往往会同时触发多个后续操作，比如
      用户活跃度统计、消息通知、排行榜更新等。
    • 如果这些逻辑都放在接口里同步处理，
      接口会变得非常臃肿，而且一旦某个功能出问题，
      就可能影响整个请求的响应。
    • 通过引入 MQ，把这些操作拆成事件驱动的方式，
      点赞接口只负责投递消息，
      后续逻辑由不同消费者异步处理，
      这样在解耦、扩展性和稳定性上都会更好一些。

第三问(致命高频)

那我继续问你：

RabbitMQ 消息一定不会丢吗？
如果消费者处理失败了，你们是怎么处理的？

  • 答:
    • 在使用 RabbitMQ 的时候，消息并不是绝对不丢的，
      所以我们在设计时是按照 “可能失败” 来考虑的。
    • 在发送端，可以通过开启 消息持久化 和 确认机制，
      确保消息至少成功投递到 Broker；
    • 在消费端，如果业务处理失败，会通过 重新入队或重试机制，
      避免消息直接丢失；
    • 同时对于一些关键业务，我们也会做好日志记录，
      即使出现异常，也能通过补偿或人工介入进行修复。

第四问(终结拷打)

如果 RabbitMQ 挂了，
你这个“点赞 / 评论”功能还能不能用？
你们是怎么考虑这个问题的？

  • 答:
    • 还能用
    • 在设计时我们就考虑过 RabbitMQ 不可用的情况，
      所以目标是 保证核心功能不中断，同时尽量不丢消息。
    • 在 MQ 异常时，我们会先进行降级处理，
      将需要投递的消息先写入本地消息表持久化到业务数据库中，
      点赞、评论这些核心操作本身仍然可以正常完成。
    • 后续通过定时任务对未发送成功的消息进行重试，
      在 MQ 恢复后再补发消息，
      从而保证系统在异常情况下的可用性和最终一致性。

第五问(Redis+排行榜)

你简历里写到：
“结合 Redis 的 ZSet 结构与幂等校验机制，实现高性能的用户活跃度排行榜”

你这个“用户活跃度”是怎么定义的？
点赞、评论、收藏权重一样吗？
如果我疯狂点赞再取消，会不会刷榜？

  • 答:
    • 用户活跃度是一个综合指标，
      包括连续登录天数以及用户的核心行为，
      不同的行为对应不同的权重，比如评论的权重会高于点赞。
    • 排行榜是基于 Redis 的 ZSet 实现的，
      每次用户产生有效行为时，会对对应的 score 做增量更新。
    • 针对重复操作或者刷榜的问题，我们在更新 score 前做了幂等校验，
      比如点赞这类行为，会先判断用户是否已经处于“已点赞”状态，
      只有状态发生变化时，才会对 ZSet 的 score 进行更新，
      从而避免通过频繁地取消和重复操作来刷榜。

第六问

你这个排行榜用的是 Redis，
那如果 Redis 重启或者数据丢了，
排行榜还能恢复吗？
你们有没有考虑过这个问题？

  • 答:
    • 排行榜数据主要存放在 Redis 中，
      为了降低数据丢失风险，我们开启了 RDB 和 AOF 的混合持久化。
    • 但在设计上我们并没有把 Redis 当作唯一的数据源，
      核心的用户行为数据仍然会落库，
      Redis 更多是作为高性能的实时排行榜使用。
    • 在极端情况下如果 Redis 数据异常，
      可以通过数据库中的行为数据进行重建，
      只是实时性会有所下降，但不会影响业务的正确性。

第七问

那如果你删 Redis 的时候失败了，
或者删成功后立刻又有请求把旧数据写回缓存，
你这个排行榜或业务数据会不会出问题？
你是怎么接受或处理这种情况的？

  • 答:
    • 这种场景在高并发下确实是有可能出现问题的，
      比如删除缓存失败，或者在并发读写下出现短暂的不一致。
    • 在当时的业务场景下，我们并没有追求强一致，
      而是选择了先写数据库、再删缓存来保证最终一致性，
      并接受在极端并发下存在短时间的数据不一致。
    • 对于我们这个业务来说，
      比如排行榜或活跃度数据，
      短暂的不一致是可以接受的，
      后续通过缓存失效或数据重建可以恢复正确状态。

第八问(经典拷打)

如果让你现在重构这个项目，
你最想改动的一点是什么？为什么？

  • 答:
    • 如果现在让我回头重构这个项目，
      我会考虑将整体架构从单体的 Spring Boot，
      向 Spring Cloud 方向拆分，
      把用户、内容、互动、消息等模块进行解耦，
      提升系统的扩展性和可维护性。
    • 当然，当时的业务规模和我个人能力还不足以支撑完整的微服务架构，
      所以项目阶段选择了单体实现；
      目前我对 Spring Cloud 还处在理解和学习阶段，
      后续如果业务复杂度和并发量上来，
      这是我会重点补强和演进的方向。

第九问(jwt令牌)

你们为什么选择 JWT？
不用 Session + Cookie 行不行？

  • 答:
    • 我们选择 JWT，主要是考虑到项目是前后端分离架构，
      使用 Token 的方式更符合这种场景，
      同时 JWT 是无状态的，不需要在服务端维护会话信息，
      对后续的扩展和部署会更友好。
    • 当然 JWT 也不是完全没有问题，
      比如无法主动失效、需要考虑过期和安全性，
      但在当前项目规模和业务场景下，这个取舍是可以接受的。

第十问

JWT 是放在 Cookie 里，
还是放在请求头里的？
你们为什么这么选？

  • 答:
    • 我们是把 JWT 放在请求头中进行传递的，
      主要是因为项目是前后端分离架构，
      使用 Header 方式对跨域和多端接入更友好。
    • 如果使用 Cookie，需要额外考虑跨域配置以及 CSRF 风险，
      而通过请求头携带 Token，可以避免浏览器自动携带 Cookie 带来的问题，
      同时也让认证逻辑更加清晰、可控。
    • 在当前项目规模下，这种方式在开发效率和安全性之间是一个比较合适的选择。

第十一问

JWT 一旦发出去，在过期之前是不是就“永远有效”？
如果用户主动退出登录，
或者 Token 被别人拿到了，你们怎么办？

  • 答:
    • JWT 本身是无状态的，一旦发放，在过期之前确实是有效的，
      所以我们主要是通过设置合理的过期时间来控制风险，
      避免 Token 长时间有效。
    • 对于用户主动退出登录的场景，我们会在前端清除 Token，
      服务端依赖过期机制来保证安全。
    • 至于 Token 被窃取的问题，这是 JWT 的一个天然风险，
      在当前项目规模下我们没有引入黑名单或双 Token 机制，
      而是通过 HTTPS、较短的有效期来降低风险。
    • 如果后续对安全性要求更高，
      可以考虑引入 Refresh Token 或服务端 Token 黑名单来增强控制能力。

第十二问

为什么要用 ThreadLocal？
不用每个接口都传 userId 行不行？

  • 答:
    • 使用 ThreadLocal 的主要目的，是在一次请求的处理线程中，
      存储当前登录用户的上下文信息，
      这样在业务层就不需要在方法参数中层层传递 userId。
    • 用户信息在登录校验拦截器中解析并放入 ThreadLocal，
      业务代码只需要关注业务本身，
      不需要感知登录校验的细节，
      从而实现认证逻辑与业务逻辑的解耦。
    • 同时 ThreadLocal 是基于线程隔离的，
      在一次请求对应一个线程的模型下，
      可以安全地存储当前用户信息，
      用来统一当前请求的上下文。

前言JVM我自己觉得也是进阶篇的内容了,一般的程序不会涉及到jvm调优这些的,但是想要成为中级甚至是高级程序员,JVM应该是必不可少的一部分,这里就只简单聊一下 内存管理简单说一下JVM的内存区域JVM的内存区域可以分为:程序计数器,虚拟机栈,本地方法栈,堆,方法区 其中堆,方法区的线程是共享的,程序计数器,虚拟机栈,本地方法栈的线程是私有的 程序计数器:程序计数器也被称为 PC 寄存器,是一块较小的内存空间;它可以看作是当前线程所执行的字节码行号指示器 虚拟机栈:Java 虚拟机栈的生命周期与线程相同 对象创建的过程了解吗当我们使用 new 关键字创建一个对象时: JVM 首先会检查 new 指令的参数是否能在常量池中定位到类的符号引用, 然后检查这个符号引用代表的类是否已被加载、解析和初始化。如果没有,就先执行类加载; 如果已经加载,JVM 会为对象分配内存完成初始化,比如数值类型的成员变量初始值是 0,布尔类型是 false,对象类型是 null; 接下来会设置对象头,里面包含了对象是哪个类的实例、对象的哈希码、对象的 GC 分代年龄等信息; 最后,JVM 会执行构造方法...
